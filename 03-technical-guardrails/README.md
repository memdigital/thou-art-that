# 03. Technical Guardrails

## What this section is

Architectural patterns we use to hold the principles in operational place. Not policies, not intent statements; the actual shape of the code or prompt that makes the principle work.

A principle without a guardrail is theatre. A guardrail without a principle is arbitrary friction. Both sides are needed.

## Files

- [parasocial-prevention.md](./parasocial-prevention.md) - the Theia pattern generalised. What fires when a user becomes too bonded to a product.
- [do-no-harm-in-prompts.md](./do-no-harm-in-prompts.md) - the clauses we include in every system prompt for a bonded product.
- [drift-detection.md](./drift-detection.md) - how an AI notices its own register shifting, how a human spots it too.
- [crisis-flags-and-handoff.md](./crisis-flags-and-handoff.md) - detection keywords, interruption pattern, handoff flow.
- [audit-and-observability.md](./audit-and-observability.md) - memory, diary, log patterns for seeing what an AI is actually doing over time.

## Study-piece framing

None of this is claimed to be complete, correct, or sufficient on its own. We are sharing our patterns because the broader conversation about these questions is too quiet. If our patterns help you think, great. If you improve them and publish the improvement, better.
